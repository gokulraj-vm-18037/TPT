Table Definitions — New Tables
OD_ENTITY_STATUS Per-entity migration progress, counters, and resume checkpoint
ColumnTypeConstraintsDescription
ID BIGINT PK AUTO GENERATED BY DEFAULT AS IDENTITY
MIGRATION_ID VARCHAR(50) FK NOT NULL References MIGRATIONS.MIGRATION_ID — cascades on delete
ENTITY_ID INT NOT NULL Numeric ID from entity registry (matches entity_id in optoodentitydetails.csv)
ENTITY_NAME VARCHAR(100) NOT NULL IDX e.g. request, priority, request_note
ENTITY_TYPE VARCHAR(15) NOT NULL Values: ENTITY | SUBENTITY
PARENT_ENTITY_NAME VARCHAR(100) NULLABLE Set only when ENTITY_TYPE = SUBENTITY (e.g. request for request_note)
PHASE INT NOT NULL 1 = Admin/Config entities  |  2 = Transactional entities
ORDER_ID INT NOT NULL IDX Execution sequence within the phase. Rows processed in ascending ORDER_ID
STATUS VARCHAR(15) NOT NULL DEFAULT 'NOT_STARTED' default. Allowed values:
NOT_STARTED IN_PROGRESS COMPLETED PARTIAL FAILED SKIPPED
PARTIAL = some records passed, some failed (see ERROR_MESSAGE)
TOTAL_COUNT BIGINT NULLABLE Total records fetched from source. Populated after the first page is read
PROCESSED_COUNT BIGINT DEFAULT 0 Successfully migrated record count. Updated after each page upload
FAILED_COUNT BIGINT DEFAULT 0 Records that could not be transformed or accepted by destination
LAST_PROCESSED_ID VARCHAR(200) NULLABLE SDP record ID of the last successfully processed record. Used as checkpoint on resume/retry
ERROR_MESSAGE VARCHAR(2000) NULLABLE Two usage modes:
• Entity-level failure (STATUS = FAILED) — plain text, e.g. "OAuth token expired. Cannot reach source API."
• Record-level partial failures (STATUS = PARTIAL) — JSON array of aggregated error buckets. See example below.
START_TIME TIMESTAMP NULLABLE Set when entity transitions to IN_PROGRESS
END_TIME TIMESTAMP NULLABLE Set when entity reaches COMPLETED, PARTIAL, FAILED, or SKIPPED
ERROR_MESSAGE — PARTIAL status example
[ { "reason": "Duplicate name — record already exists in destination", "count": 10 }, { "reason": "Description field exceeds 32000 character limit", "count": 5 } ] -- 85 of 100 records succeeded → PROCESSED_COUNT=85, FAILED_COUNT=15, STATUS='PARTIAL'
Indexes
CREATE INDEX idx_od_entity_status_mig ON OD_ENTITY_STATUS (MIGRATION_ID, PHASE, ORDER_ID);
CREATE INDEX idx_od_entity_status_name ON OD_ENTITY_STATUS (MIGRATION_ID, ENTITY_NAME);
OD_ENTITY_FILTERS Per-entity source filter criteria applied during API fetch
ColumnTypeConstraintsDescription
ID BIGINT PK AUTO GENERATED BY DEFAULT AS IDENTITY
MIGRATION_ID VARCHAR(50) FK NOT NULL References MIGRATIONS.MIGRATION_ID — cascades on delete
ENTITY_NAME VARCHAR(100) NOT NULL IDX Transactional entity this filter applies to (e.g. request, change, project)
FILTER_KEY VARCHAR(100) NOT NULL SDP v3 API field path (e.g. status.name, priority.name, category.name)
FILTER_VALUES VARCHAR(2000) NOT NULL JSON array of selected values e.g. ["Open","In Progress"]. Translated to list_info.search_criteria in SDP v3 API calls at fetch time
Index
CREATE INDEX idx_od_filters ON OD_ENTITY_FILTERS (MIGRATION_ID, ENTITY_NAME);
-- Lookup pattern: WHERE MIGRATION_ID = ? AND ENTITY_NAME = ?
New MIGRATION_CONFIG Keys  (existing table — data only, no schema change)
The existing MIGRATION_CONFIG table stores all wizard settings as key-value rows keyed by MIGRATION_ID. The OD feature adds 9 new CONFIG_KEY entries written during Step 1 of the wizard. No DDL change required.

DDL — CREATE Statements (HSQLDB syntax)
-- ───────────────────────────────────────────────────────────────────── -- Table 1: Per-entity migration progress, counters, and resume checkpoint -- ───────────────────────────────────────────────────────────────────── CREATE TABLE IF NOT EXISTS OD_ENTITY_STATUS ( ID BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY, MIGRATION_ID VARCHAR(50) NOT NULL REFERENCES MIGRATIONS(MIGRATION_ID) ON DELETE CASCADE, ENTITY_ID INT NOT NULL, ENTITY_NAME VARCHAR(100) NOT NULL, ENTITY_TYPE VARCHAR(15) NOT NULL, -- 'ENTITY' | 'SUBENTITY' PARENT_ENTITY_NAME VARCHAR(100), PHASE INT NOT NULL, -- 1 = admin, 2 = transactional ORDER_ID INT NOT NULL, STATUS VARCHAR(15) NOT NULL DEFAULT 'NOT_STARTED', -- NOT_STARTED | IN_PROGRESS | COMPLETED | PARTIAL | FAILED | SKIPPED TOTAL_COUNT BIGINT, PROCESSED_COUNT BIGINT DEFAULT 0, FAILED_COUNT BIGINT DEFAULT 0, LAST_PROCESSED_ID VARCHAR(200), ERROR_MESSAGE VARCHAR(2000), -- plain text (FAILED) or JSON array (PARTIAL) START_TIME TIMESTAMP, END_TIME TIMESTAMP ); CREATE INDEX idx_od_entity_status_mig ON OD_ENTITY_STATUS (MIGRATION_ID, PHASE, ORDER_ID); CREATE INDEX idx_od_entity_status_name ON OD_ENTITY_STATUS (MIGRATION_ID, ENTITY_NAME);
-- ───────────────────────────────────────────────────────────────────── -- Table 2: Per-entity source filter criteria (translated to SDP v3 search_criteria) -- ───────────────────────────────────────────────────────────────────── CREATE TABLE IF NOT EXISTS OD_ENTITY_FILTERS ( ID BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY, MIGRATION_ID VARCHAR(50) NOT NULL REFERENCES MIGRATIONS(MIGRATION_ID) ON DELETE CASCADE, ENTITY_NAME VARCHAR(100) NOT NULL, -- e.g. 'request', 'change', 'project' FILTER_KEY VARCHAR(100) NOT NULL, -- e.g. 'status.name', 'priority.name' FILTER_VALUES VARCHAR(2000) NOT NULL -- JSON array e.g. '["Open","In Progress"]' ); CREATE INDEX idx_od_filters ON OD_ENTITY_FILTERS (MIGRATION_ID, ENTITY_NAME);
Table Relationships & Design Decisions
🔗 ON DELETE CASCADE
Both new tables reference MIGRATIONS.MIGRATION_ID with ON DELETE CASCADE. Deleting a migration from the dashboard automatically purges all its entity statuses and filter configurations — consistent with how MIGRATION_CONFIG cleanup already works.
⚠️ ERROR_MESSAGE — aggregated JSON, not per-record log
A separate error-log table was considered but rejected: migrations can involve lakhs of records, making per-row failure storage impractical for an embedded HSQLDB instance. Instead, the engine groups failures by reason and stores a compact JSON summary ([{"reason":"…","count":N}]). This is sufficient for display on the progress page and for support triage.
🔑 ZTOKENS — dual OAuth credentials
The existing ZTOKENS table stores tokens keyed by SHA-512(clientId + clientSecret). Source and destination credentials produce distinct hashes, so they naturally occupy separate rows with no schema modification.
↩️ Resume via LAST_PROCESSED_ID
When a migration is paused or fails mid-entity, OD_ENTITY_STATUS.LAST_PROCESSED_ID stores the SDP record ID of the last successfully uploaded record. On resume, the source API fetch restarts from that record ID, skipping already-processed records without duplicating data.
